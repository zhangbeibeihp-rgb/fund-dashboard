🔴 P0 — 阻断问题（必须修复才能上线）

这些问题会导致功能崩溃、安全漏洞或数据丢失
P0
createScenarioChart 引用未定义变量 `goal`
初始化时 createCharts() → createScenarioChart() 内引用 goal.monthly，但 goal 在该作用域未定义（应为 goals[currentGoalId].monthly）。导致 ReferenceError，情景模拟图无法渲染，后续图表初始化也可能连锁失败。
Line 2305: calcFV(totalAsset, goal.monthly, r, y)
P0
XSS 漏洞 — 用户输入未转义直接注入 innerHTML
多处使用 innerHTML + 模板字符串注入用户输入。目标名称、账号名称等字段未做 HTML 转义。恶意输入如 <img src=x onerror=alert(1)> 会执行任意 JS 代码。涉及函数：renderGoalTabs()、confirmAddAccount()、renderAIPredictions() 等。
Line 1777, 3034, 2838 等多处 innerHTML 赋值
P0
无后端 API — 所有数据硬编码
15支基金持仓、27支推荐池、15条AI预测、8笔交易记录全部硬编码在 JS 中。无实时净值获取、无交易接口、无数据存储。产品无法反映真实持仓变化，用户无法修改持仓数据。
Line 1785-1812 (funds), 2472-2570 (fundPool), 2790-2806 (aiPredictions)
P0
无用户认证系统
无登录注册、无 Session/Token、无多用户数据隔离。任何人打开同一浏览器即可查看/修改全部数据。涉及金融数据的产品必须有用户认证和权限控制。
全局 — 无 auth 模块
P0
switchAccount 逻辑 Bug — 三元表达式两边相同
inputRetireAge.value = acct.age ? 60 : 60; 三元表达式 true 和 false 分支都是 60，永远返回 60。应根据目标类型或账号配置动态计算目标达成年龄。
Line 2965
P0
多账号系统不完整 — 新账号无独立基金数据
新增账号后 totalAsset=0、fundCount=0，但所有图表/持仓/再平衡仍显示第一个账号的硬编码数据。switchAccount() 仅更新右侧卡片和目标输入，未切换基金数据源。多账号功能形同虚设。
Line 2935-2969 (switchAccount), 3012-3039 (confirmAddAccount)

🟡 P1 — 高优先级（上线前应修复）

影响用户体验和数据完整性
P1
截图上传功能未实现
加减仓记录页的上传区域（Line 1108）仅 HTML 结构，无 JavaScript 事件绑定。点击/拖拽无任何反应。这是一个已展示给用户但无法使用的功能入口。
Line 1108-1111 (.upload-area 无 onclick/onchange)
P1
KPI 卡片数据硬编码不更新
总览页6个 KPI 卡片（总资产¥10,920、昨日收益+¥376.87、持有收益+¥736.90等）全部为静态 HTML 硬编码。切换账号或目标后不更新。AI 预测 KPI 虽有 JS 更新但初始值也是硬编码。
Line 714-744 (KPI cards)
P1
CDN 单点故障 — Chart.js 依赖外部 CDN
Chart.js 从 jsdelivr CDN 加载。CDN 宕机或网络受限（如国内访问不稳定）时，全部图表无法渲染，仪表盘核心功能瘫痪。应本地化部署或使用多 CDN 回退。
Line 7: <script src="https://cdn.jsdelivr.net/npm/chart.js...">
P1
alert() 做用户反馈 — 体验差
多处使用 alert() 弹窗做用户反馈（addFundToWatch、showFundDetail、generateAIAdvice 等）。alert 阻塞 UI、无法自定义样式、移动端体验极差。应改用 Toast/Modal 组件。
Line 2712, 2720, 3051, 1745 等
P1
无数据导出/备份机制
用户的目标设定存储在 localStorage，但无法导出备份。清除浏览器数据后全部丢失。金融类产品必须有数据导出和备份功能。
localStorage key: fund_goals_v7 (Line 1700)
P1
Chart 实例创建无错误处理
createCharts() 中连续创建多个 Chart 实例，无 try-catch。任意一个图表创建失败（如 canvas 不存在）会导致后续代码不执行，连锁崩溃。
Line 2346-2433 (createCharts)

🔵 P2 — 中优先级（上线后迭代改进）

P2
无 PWA / 离线支持
无 Service Worker、无 manifest.json。断网后完全不可用。作为金融工具类应用，离线查看持仓是基础需求。
P2
无加载状态 / 骨架屏
所有渲染同步执行。数据量大时（如基金推荐池27支卡片渲染）会出现短暂卡顿，无加载指示器。
P2
无可访问性 (a11y)
无 ARIA 标签、无键盘导航支持、无屏幕阅读器适配、图表无文字替代。不符合 WCAG 2.1 AA 标准。
P2
移动端表格溢出
持仓明细表（11列）、预估明细表（11列）在移动端横向溢出严重，虽有 table-wrap 滚动但操作体验差。
P2
localStorage 无版本迁移策略
key 为 fund_goals_v7，如果数据结构升级（如 v8），旧数据无法自动迁移。需要版本检测和数据迁移函数。

⚪ P3 — 建议改进（长期技术债）

P3
单文件架构 — 3313行全在一个 HTML
CSS、HTML、JS 全在一个文件中。难以协作开发、无法做代码审查、无 Tree-shaking。应拆分为模块化项目（Vue/React + 组件化）。
P3
无构建系统
无 Webpack/Vite 打包、无代码压缩、无 CSS 提取。生产环境应使用构建工具优化。
P3
无 TypeScript — 纯 JS 无类型检查
funds、goals、accounts 等数据结构无类型定义。重构时极易引入类型相关 bug。
P3
无测试覆盖
calcFV、calcRequiredMonthly、calcYearsToGoal 等核心复利计算函数无单元测试。财务计算正确性无保障。

📋 最终审查结论

结论：当前版本不可上线。

项目作为个人 Demo / 原型质量不错 — 10 个功能模块覆盖了基金组合管理的核心链路，UI 设计专业，QDII 净值披露机制说明详实，AI 预测和推荐池模型有理论支撑。但从产品上线标准来看，存在 6 个 P0 阻断问题（代码 bug、安全漏洞、无后端、无认证），以及大量功能缺失。

核心差距：

  • 数据层：100% 硬编码，无 API、无数据库、无实时数据
  • 安全层：XSS 漏洞、无认证、无加密、无 HTTPS
  • 功能层：截图上传未实现、多账号不完整、KPI 不更新
  • 合规层：无金融资质、无风险测评、无用户协议
  • 工程层：单文件 3313 行、无测试、无构建系统

APP 化建议：当前 HTML 单页面可通过 PWA 技术快速封装为"准 APP"（可安装到手机桌面），但要成为真正的金融类 APP（应用商店上架），需要完成后端、认证、合规三大基础建设。建议按上方路线图分 5 个阶段推进，预计总工期 10-17 周。
